Research Brief: Adaptive Fusion and Verification

Summary

The highest-value implementation is not a generic fusion command. It is a pre-build behavioral contract, an authoritative bounded builder-feedback loop, disagreement-aware review receipts, decision-leverage policy, and per-attempt economics. Existing browser recovery, retry policy, builder continuity, routing, cleanup, shadow parity, and optimized review tiers should be reused rather than rebuilt.

Multi-model synthesis should remain selective. High uncertainty warrants an additional independent opinion; high consequence warrants stronger independent verification. Provider selection remains governed separately by task fit, sensitive paths, live-tool needs, availability, and quality floors.

Project Context

  • origin/main is the correct base at 21f3d9b. It already contains the Codex/OpenRouter cascade, granular routing, focused ordinary-chunk review, batched shadow observation, bounded run size, active-versus-waiting metrics, and other speed work.
  • Prior measured runs showed substantial orchestration and waiting overhead while provider/token latency telemetry was missing. The project intentionally optimized repeated review and shadow work without removing sensitive-path or final-review gates.
  • The workflow kernel was deliberately introduced as deterministic, testable state/evidence machinery. Pipeline remains authoritative; kernel events validate, replay, compare, and measure the authoritative workflow.
  • The known full-validation failure is an external assembly-baseplate UX declaration/parser mismatch. Depot-owned focused checks pass; this sibling-repository coupling should not be silently repaired as part of the feature unless it becomes a direct compatibility requirement.
  • ai-memory MCP and RAG were unavailable. Project-history research used the local Codex memory registry and rollout summaries as a read-only adapter. Context7 was not applicable because no third-party runtime library API is being introduced.

Domain Knowledge

Behavioral contract

Add a general behavioral contract beside the browser-specific VerificationProfile. It should use stable requirement IDs, prohibited regressions, deterministic commands encoded as argv arrays, requirement-to-check coverage, existing persona/browser case IDs, explicit unverifiable requirements, baseline expectation, revision, previous digest, and revision justification.

The protected authority is the contract digest and required evidence—not test file contents. Builders may add or improve tests. Weakening or removing a behavioral requirement requires an append-only revision event, bounded justification, and human approval evidence.

Authoritative feedback

After deterministic validation fails, send the bound contract digest, failing check IDs, safe evidence references, stable failure signature, and remaining retry budget to the same builder session when the host can prove continuity. Otherwise dispatch a replacement with an explicit continuity-loss receipt. Identical-signature or exhausted failures stop with human help rather than spawning open-ended retries.

Disagreement-aware convergence

Extend the existing consolidator and output contract. Preserve canonical finding IDs, corroborated sources, unique retained findings, disputed positions, retained rationale, and merged/discarded reason codes. Keep existing todo filenames and source_agents for compatibility; add a separate finding_contribution receipt stage for structured metrics.

Decision leverage

Add an approved decisionProfile containing uncertainty, consequence, and rationale. Do not overload workflow class, safety risk, overlap risk, complexity, executor, or routing override. Do not persist a derived tier; derive workflow depth from a single routing-policy section.

Design References

  • Evaluate the required end state and explicit safety checkpoints rather than prescribing one model trajectory.
  • Use typed, bounded handoffs with objectives, allowed scope, artifact references, expected output shape, and terminal status.
  • Keep synthesis provenance linked to immutable source artifacts; synthesis never replaces raw reviewer outputs.
  • Scale additional perspectives to uncertainty, consequence, and actual parallelizability. Avoid default debate.
  • Use deterministic checks as proof. Model review is calibrated evidence, not correctness authority.
  • Measure retained marginal value per dollar/minute, not raw agent activity or finding count.

  • OpenAI Evals guide: separates test-data schema from testing criteria.
  • Anthropic multi-agent research system: durable artifacts, bounded delegation, end-state evaluation, scaling cautions, and high token overhead.
  • OpenAI Agents SDK handoffs: typed handoff inputs, filtering, and explicit transfer boundaries.

Caveats: Anthropic's published gains and approximately 15× token use are research-system results, not coding forecasts. OpenTelemetry GenAI conventions are evolving, so Depot should adopt versioned concepts without adding a runtime dependency. Missing provider usage must remain unknown rather than becoming zero. Raw content tracing remains local, redacted, and artifact-referenced by default.

Constraints and Risks

  • Prediction authority assumes run.initialized → independent_prediction_bound → run.started. Contract binding must not be inserted before run.started without a broader compatibility change.
  • The contract schema should be independently versioned. Do not bump the core event schema or add contract fields to RunState unnecessarily.
  • New manifests can require decisionProfile and a contract binding, but legacy manifests must preserve current behavior and record default provenance.
  • Attempt usage plus run-total usage can double-count. Explicit scope and precedence rules are required.
  • unique_reviewer_yield currently counts reviewers with findings, not unique retained findings. Preserve it and add accurately named metrics.
  • High uncertainty should add synthesis; high consequence should add verification. Neither may override sensitive-path or trusted-provider rules.
  • Do not restore full multi-agent review for ordinary chunks or reintroduce the overhead removed in Pipeline 1.31.0.
  • Adding an eighth JSON schema and a new CLI command requires updating exact-count and behavioral CLI assertions in the workflow-kernel release validator.

Key Decisions Needed

  1. Use workflowClass: feature for this new capability set.
  2. Classify the run as uncertainty: medium, consequence: high: the desired behavior is clear, but the authoritative/shadow seam and compatibility surface are substantial.
  3. Require a behavioral contract for new verified/synthesized runs; legacy absence preserves today's standard behavior with explicit default provenance.
  4. Allow baseline expectation must_fail, may_pass, or not_runnable rather than forcing a red test for documentation or already-covered behavior.
  5. Keep sibling-repository UX compatibility outside the feature unless implementation changes the parser contract or release validation proves the base is newly incompatible.
  6. Version the expected release as workflow-kernel 0.3.0, Pipeline 1.32.0, and a compatible dm-review minor bump, recalculated after refreshing the base.
