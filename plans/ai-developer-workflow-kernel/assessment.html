Assessment: AI Developer Workflow Kernel

Key Requirements (cached from original-prompt.md)

  1. Deliver the complete workflow-kernel improvement set rather than one isolated recommendation.
  2. Use the literal Depot pipeline with design, assessment, research, planning, adversarial review, execution, dm-review, verification, and cleanup gates.
  3. Decompose delivery into safe, testable slices while preserving existing pipeline and dm-review compatibility.
  4. Build an executable control plane rather than adding orchestration prose alone.
  5. Preserve Claude/Codex behavior, provider security boundaries, evidence receipts, and honest degradation.
  6. Validate transitions and failure paths behaviorally, not only with text-contract checks.
  7. Deliver a clean feature branch with requirement-level evidence.
  8. Use an additive Python standard-library kernel with shadow, enforce, and opt-in native modes.
  9. Keep the epic repo-local; do not add a daemon or external ticket ingestion.
  10. Track and clean owned Git and Docker resources; automatically remove only Depot-labeled Docker resources older than 24 hours.
  11. Discover project UX suites and make declared personas blocking for UI/integration completion.
  12. On required-browser failure, capture evidence, restart the primary browser, try another browser engine, then require human help.

Code State

Architecture

Depot already has most of the conceptual pieces of a workflow engine, but they are split among Markdown command contracts, agent definitions, JSON routing policy, shell dispatchers, generated aliases, HTML/JSON planning artifacts, and receipts. The central execution contract is plugins/pipeline/agents/workflow/execution-orchestrator.md, a 1,589-line prose state machine that coordinates manifest validation, host/provider routing, worktree isolation, validation, review, visual proof, merging, postmortems, and cleanup.

The executable layer is strongest around narrow deterministic problems: provider cascade descent, manifest generation, JSON checks, generated-shim validation, and selected shell portability cases. It is weakest at cross-phase state, evidence-bound transitions, resource ownership beyond Git, and replayable failure behavior. dm-review has a mature lane model and explicit coverage accounting, but its orchestration remains another large interpreted Markdown contract.

What is working

  • Policy separation: routing policy, model cascade, and harness profiles already distinguish task fit, fallback order, and host capabilities.
  • Honest degradation: dm-review names failed or unavailable lanes and prevents a missing core reviewer from reading as clean.
  • Durable artifacts: manifests, Airlift checkpoints, receipts, todos/issues, campaign state, and postmortems provide several useful resumption anchors.
  • Git cleanup: immediate registration, ownership boundaries, merge proof, and blocked-removal reporting are unusually mature.
  • Provider security: sensitive-path routing overrides cheaper provider preferences.
  • Persona knowledge: assessment, promptcraft, orchestrator, and reviewers already understand tests/ux/; the missing piece is a single blocking execution contract.
  • Browser recovery precedent: the visual browser reviewer already closes/restarts the browser, tries another tool surface, and stops visibly when all attempts fail.

Tech debt and gaps

  • God contract: the 1,589-line execution orchestrator mixes state, policy, shell procedures, receipts, recovery, and prose. Changes are difficult to test in isolation.
  • Inconsistent browser authority: one reviewer has a recovery ladder, while pipeline preflight can degrade to curl or defer to the caller. The same failure can produce different outcomes by entry point.
  • Advisory persona coverage: current rules ask for two or three persona lenses but do not prove every declared required persona/scenario executed.
  • No builder continuity: failed validation marks a chunk failed and skips dependents; no attempt stores a resumable worker handle.
  • No Docker ownership: Docker is required for verification and influences isolation mode, but containers, networks, and volumes are not registered, labeled, inventoried, or cleaned by the workflow.
  • Fragmented durable state: several artifacts record partial state, but no versioned append-only event model can reconstruct a run.
  • Text-heavy validation: important validators assert that required phrases exist, not that failure transitions actually occur.
  • Bootstrap risk: pipeline will modify its own execution contract. New enforcement cannot become authoritative until old-path shadow comparisons and receipt replay pass.

Dependencies

  • Internal: pipeline commands, promptcraft manifests, dm-review lanes, cleanup contract, Airlift, generated command skills, provider routing files, HTML artifact helpers, and composition validators.
  • External runtime: Python 3 standard library, Git, optional Docker, browser tool surfaces, Codex/Claude host dispatch, and optional OpenRouter/DeepSeek wrappers.
  • Reverse dependencies: every project invoking pipeline or dm-review, especially Assembly and assembly-baseplate with Docker-only verification and tests/ux/ personas.
  • Cycle risk: dm-review cannot depend on pipeline as a plugin. Shared kernel policy must live in a neutral executable/reference boundary or be consumed optionally.

Assessment coverage

Degraded fan-out: the environment selected OpenRouter for read-heavy assessment, but workspace-data policy rejected sending private repository architecture to that external provider. The independent native assessment worker also failed to return within its bounded assessment window and was stopped. Consolidation therefore uses direct local inspection, repository history, existing memory evidence, and exact file/line checks. No external assessment output is represented as completed.

UX State

Skipped — no Depot application UI or dev server. This feature changes orchestration policy and tooling. Existing browser and persona rules were inspected as code; no rendered product baseline exists in this repository.

Test Personas

Depot has no local tests/ux/ suite or dev-mode auth personas. Downstream Assembly projects do. The kernel must discover their project-local persona indexes, task files, coverage matrices, auth field names, browsers, and viewports without copying secret values into artifacts.

Recent Lessons That May Apply

  • Host adapters must preserve gates: the Codex-native adapter succeeded because it mapped host capabilities without weakening worktrees, review, cleanup, memory, or receipts.
  • Runtime staleness is distinct from source correctness: Assembly browser output has remained stale behind old container child processes; browser recovery must check runtime/container state before changing product code.
  • Cross-browser and role proof matter: durable UX verification has required both Chrome/Chromium and Firefox plus privileged and ordinary-member flows.
  • Resume from objective artifacts: browser locks, worktree failures, orchestration stalls, and usage limits are recoverable when manifests, worktrees, checkpoints, and receipts are preserved.

Key Findings

  1. The design should wrap before replacing. Manifests, routing policy, cascade dispatch, cleanup, and receipts are proven primitives. Slice 1 should observe them through adapters rather than rewrite them.
  2. The orchestrator is the extraction map. Its size and mixed responsibilities justify a kernel, but also define a detailed parity checklist.
  3. Browser and persona behavior needs one authority. Strong rules already exist, but inconsistent entry points allow silent degradation or partial coverage.
  4. Docker cleanup and session continuity are genuine missing capabilities. They are not currently hidden elsewhere in the system.
  5. The four-slice order is correct. Foundation must precede integration; integration must emit events before dm-review/UX can consume them; hardening and promotion must remain last.

Recommendations

  1. Define the smallest versioned run/event schema and legal-transition engine first, with atomic storage and replay tests.
  2. Build explicit adapters around the current manifest, cascade dispatcher, Git cleanup contract, and receipt formats before moving any authority.
  3. Create a shared browser recovery policy and verificationProfile before changing downstream persona rules.
  4. Add Docker ownership as a sibling contract to Git cleanup, including labels, registry, 24-hour TTL sweep, and blocked-removal evidence.
  5. Use shadow receipt replay as the bootstrap gate; no enforce/native transition should depend on untested new behavior.
