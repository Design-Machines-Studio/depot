AI Developer Workflow Kernel

Evidence supports an additive, standard-library control plane that wraps the existing pipeline and dm-review contracts before it enforces them. The new kernel should unify state, ownership, retries, persona proof, and browser recovery while leaving current manifests, routing, receipts, and host adapters authoritative during shadow mode.

Research Answer

Confirmed

Adapter-first is the safe architecture

Current manifests, provider routing, Git cleanup, evaluation receipts, Codex shims, and review convergence are proven contracts. Translate them into kernel events and replay their receipts before moving authority.

Confirmed

The four-slice sequence holds

Build core state and replay first, resource ownership second, browser/persona verification third, and workflow adapters plus promotion controls fourth.

Correction

Docker cleanup needs typed strategies

Containers and networks accept label plus age filters. Volumes accept label filters but no until filter, so stale owned volumes must be listed, inspected, age-checked, and removed individually.

New contract

Atomic write is not concurrency control

Same-directory replacement protects file integrity. A run-scoped single-writer lease and revision check are still required to prevent lost state updates.

External Contract Checks

Crash-safe local state

Python documents os.replace as atomic on success and warns that cross-filesystem replacement may fail. The state writer should therefore create its temporary file in the destination directory, write deterministic UTF-8 JSON, flush, call os.fsync, replace the target, and best-effort fsync the directory on supported Unix hosts. Recovery tests must cover interruption before and after replacement.

Owned Docker cleanup

Docker's current CLI supports conjunctive label and until filters for stopped containers and unused networks. Volume pruning supports labels but not age. The safe implementation is a resource-type adapter: filtered prune for containers/networks; label-filtered enumeration, inspect, TTL comparison, and explicit removal for unused volumes. Current-chunk resources should be cleaned at the existing end-of-chunk repository cleanup boundary; explicitly shared resources survive until their last dependent completes, and final cleanup reconciles anything blocked or interrupted. Never call broad system prune.

Browser restart and engine fallback

Playwright Test discards a failed worker process and its browser, then starts a new worker; projects run the same suite across Chromium, Firefox, and WebKit. That validates the restart and cross-engine concepts, but the browser MCP path is a different host surface. The kernel must explicitly record: initial failure evidence, close/relaunch primary attempt, alternate-engine attempt, and terminal help request. Screenshots, traces, console output, and attempt metadata belong in the evidence record.

Confirmed Gaps and Contradictions

  • No repository implementation currently provides run-state.json, events.jsonl, a workflow event schema, verificationProfile, resumable builder handle, Docker ownership label namespace, or Docker resource registry.
  • Browser behavior conflicts across entry points: one path restarts and fails visibly, another can report skipped, and pipeline can accept caller-approved curl fallback. An alternate tool surface is not necessarily a different browser engine.
  • Persona rules request two or three lenses but do not enforce completeness against a declared project matrix.
  • Builder failure currently fails the chunk and skips dependents; it does not retain and resume the original builder session.
  • Campaign state is latest-value overwrite state, not an append-only replay log.
  • Receipt enums have drifted across docs and adapters, which argues for a versioned evidence schema.
  • dm-review cannot depend directly on pipeline without a plugin cycle; shared kernel policy and schemas must live in a neutral repository surface.

Recommended Extraction Sequence

  1. Create tools/workflow_kernel/ with versioned models, append-only events, replay, legal transitions, retry policy, atomic state writes, single-writer/revision protection, fake clock, fake host adapters, tests, and a local CLI.
  2. Add a shadow manifest translator and compatibility fixtures around the current source-of-truth manifest. Compare predicted transitions and evidence to existing receipts.
  3. Generalize the Git registry pattern into a resource registry, then add Docker labels, per-chunk cleanup at the existing repository cleanup boundary, terminal reconciliation, 24-hour labeled-only sweeping, and blocked-removal evidence.
  4. Add isolation and builder-session adapters. Resume only when a host returns a real handle; otherwise emit session_resume_unavailable and use the declared terminal policy.
  5. Add a shared verificationProfile, declared-persona coverage model, and browser recovery machine. Update assessment, promptcraft, pipeline, visual review, and UX review to reference the shared contract.
  6. Integrate dm-review lanes and fallback reporting after pipeline shadow replay is stable.
  7. Wire behavioral kernel tests into full composition validation. Promote shadow to enforce only after receipt replay matches; keep native mode separately opt-in.
  8. Update canonical Claude command/plugin sources and versions, then regenerate and check Codex manifests and command-skill aliases.
