Pipeline · Phase 3 · Implementation Plan

AI Developer Workflow Kernel

Build a neutral, executable workflow kernel shared by pipeline and dm-review, then adopt it through shadow adapters before enabling enforcement. Six chunks keep the baseline runnable and make every authority change conditional on behavioral proof.

Overview

Goal

Replace repeatable orchestration prose with a deterministic, replayable control plane while preserving every current pipeline and dm-review guarantee.

Architecture

A new leaf workflow-kernel plugin owns versioned Python standard-library state, policy, adapter, and evidence contracts. Pipeline and dm-review remain host orchestrators and initially emit shadow observations.

Tech stack

Python 3 standard library, JSON/JSONL, shell validators, Git, optional Docker CLI, existing browser tool surfaces, and generated Claude/Codex plugin manifests.

Global Constraints

  • No third-party Python dependency, daemon, database, or external ticket ingestion.
  • Claude command and plugin sources remain canonical; Codex manifests and command-skill aliases remain generated.
  • Current Markdown orchestration remains authoritative in shadow mode.
  • Existing sensitive-path provider routing overrides workflow-class routing.
  • Event and verification artifacts may store auth field names, never auth values or secrets.
  • No broad Docker prune. Cleanup requires positive Depot ownership and records every disposition.
  • Chunk-owned Docker resources clean during per-chunk repository cleanup; run-shared resources survive through their last dependent; terminal cleanup reconciles residue.
  • Declared UX personas block UI/integration completion.
  • Required-browser failure must capture evidence, restart the primary browser, try another engine, then stop for help.
  • Native mode stays opt-in; changing the default requires a separate human decision.

As-built deviation and interface — Chunk 01

This block records the implemented hardening without rewriting the design-time decisions above. Exact final schema values and recursive payload scalars require exact built-in types. Reducers and writers use bounded field-wise snapshots before comparison or projection; replay returns a tuple; validation observes ledger and state under one run lease; and every ledger mutation shares the registry-backed run lease used for state publication.

command_validate: hold one RunLease across ledger validation and state comparison

command_append: require any materialization exactly matches the replay-derived state before mutation; replay reconciles missing or stale state
command_replay: after ledger validation and reconstruction, use private ledger-derived prepared issuance and the one public publication path for lease-protected authoritative replacement of missing, behind, ahead, or equal-revision divergent materialization; the revision-bound, one-shot capability is published immediately under the same lease while expected-revision CAS still rejects concurrent change
reducers and event/state writers: shared field-wise bounded snapshots before comparison or projection
state graph: nodes + dependency edges + state evidence share one aggregate item budget and one-pass node projection
public recursive text: exact scalars; 4 MiB cumulative UTF-8 budget before retained allocation
public collections: counted before allocation; bounded iterables accept at most N items but may consume one lookahead item; reconstruction limit: 100,000 events
trusted reducer updates: incrementally maintained node, edge, text, and evidence counters enforce state bounds; scalar event snapshot, recursive payload snapshot, node-update creation, graph copy/access, evidence parse/copy/membership, and dependency membership work is charged against MAX_RECONSTRUCTION_WORK=50100000; already-normalized node tuples are reused without rescanning
state publication: one public publication path derives backward permission only from closure-registry-owned immutable issuance mode; ordinary publication rejects backward revisions and retains reusable prepared capabilities; private ledger-derived prepared issuance permits CLI replay replacement in either direction without weakening lease, pinned identity, or cooperating-writer CAS checks, binds the observed expected revision, and consumes its one-shot capability before filesystem observation so success or any later failure cannot reuse it; no public rollback API exists
durable reads/writes: open descriptor must remain the authoritative exclusive pathname before success
durable parent capability: verified directory descriptor and identity own descriptor-relative open/stat/unlink/replace/fsync operations
atomic state publication: one pinned parent spans observation, temp write/fsync, revision re-observation, rename, and directory fsync
publication races: compare-and-swap and never-overwrite guarantees cover cooperating writers honoring the same RunLease; pinned checks cannot atomically exclude non-cooperating filesystem mutation in the final commit window; detected valid revision races remain RevisionConflictError
lease fork safety: child explicit release and GC close inherited descriptors without LOCK_UN; primary errors outrank cleanup errors
owned resource cleanup: temporary unlink, every descriptor close, owned handle release, and pinned-directory close are all attempted; primary failures win and sole cleanup failures are normalized
missing durable files: public missing/empty results require revalidation of the still-live pinned parent; parent disappearance or replacement is corruption
state validation: direct/from_dict share one projection; reducers use trusted construction after one public-boundary graph validation
cache resolution: candidates are checked for a runnable package in priority order and incomplete installs fall through

Rollback and Safety Boundaries

  • Shadow integration is removable without changing authoritative outcomes; disable the adapter call and retain its artifacts for diagnosis.
  • Enforcement promotion is configuration-driven and requires recorded parity evidence. Roll back to shadow by changing mode, not deleting run history.
  • Schema readers reject unknown major versions and retain original event files.
  • Cleanup never infers ownership solely from names. Missing or corrupt ownership evidence leaves the resource untouched and reports residue.
  • Native mode cannot become the default in this implementation.
