Plan: Depot Mechanical Workflow Hardening

Overview

This is a full-mode, eight-chunk feature run. It extends Workflow Kernel with bounded, independently versioned mechanical contracts; migrates Assembly, dm-review, and Pipeline onto those contracts; then closes the release surface through canonical manifests, generated Codex adapters, documentation, and validators. The implementation remains stacked on origin/codex/adaptive-fusion-verification at PR #12 head 8faeb6ddf00979451c8fd25d7f0bd97f37ae86a8.

The plan deliberately separates repository policy, derived plans, observed results, evidence identity, reviewer records, browser bundles, artifact safety, CI state, closeout state, and improvement candidates. Structured state establishes facts; Pipeline and reviewer agents retain semantic judgment. Shadow comparison and conservative legacy defaults precede any enforcement.

Execution Shape

  1. Build the shared repository-verification and evidence-binding foundation.
  2. In parallel, implement artifact safety, browser scenarios, and CI/closeout contracts against that foundation; then add incremental review records against the completed browser contract.
  3. Migrate Assembly and Pipeline/dm-review consumers after their required contracts exist.
  4. Wire all stable CLI surfaces, version/dependency changes, generated adapters, documentation, and release validators last.

Planned parallelism: after Chunk 01, Chunks 02, 03, and 05 own disjoint implementation/test files and may run concurrently. After Chunk 03, Chunks 04 and 06 may run concurrently. Chunk 07 waits for 01–06. Chunk 08 is the integration/release gate.

Chunk Details and File Ownership

01 — Repository verification foundation

Primary files: new workflow_kernel/argv.py, repository_verification.py, verification_execution.py, and evidence_binding.py; new versioned repository-profile, verification-plan, verification-result, and evidence-binding schemas; behavioral_contract.py, repository_scope.py, and adapters/git.py only where sharing existing safe argument/Git capture is required; focused test_repository_verification.py, test_verification_execution.py, and test_evidence_binding.py.

  • Keep the existing persona/browser VerificationProfile concept separate from repository verification policy.
  • Profile precedence is explicit project config, then plugin profile, then conservative unavailable; discovery may suggest but never silently override.
  • A plan binds exact profile digest, repository state, changed scope, declared risk inputs, selected/omitted lanes, reasons, expected authority, and budget.
  • A build binding represents clean or dirty work honestly and exposes a pure binding_matches-style gate.

02 — Artifact safety and staging

Primary files: new workflow_kernel/artifacts.py, artifact-classification and staging-allowlist schemas, redaction.py only for shared safe primitives, and tests/test_artifact_safety.py.

  • Lifecycle and sensitivity remain orthogonal fields.
  • Receipts record rule IDs and content digests, never the matched secret.
  • Test fixtures such as .example and deliberate secret-shape tests require explicit rules, not blanket exceptions.
  • Symlinks, hard links, traversal, oversized files, hostile filenames, and unclassified output fail closed.

03 — Browser scenarios and bundles

Primary files: new workflow_kernel/browser_scenario.py and browser_bundle.py, new scenario/bundle schemas, targeted extensions to verification.py, browser_evidence.py, and browser adapters, plus tests/test_browser_scenarios.py.

  • Add a new scenario adapter rather than widen and break the existing BrowserAdapter protocol.
  • Represent navigation, interaction, assertion, screenshot, accessibility, console, restart, alternate-engine, and human-action steps as a closed union.
  • Reuse current immutable recovery receipts and distinguish human_action_required, human_help_required, and application_failure.
  • One bundle is shared across independent interpretation lanes only while the exact build/profile/scenario binding remains current.

04 — Read-only review and incremental findings

Primary files: new Kernel review-record module and finding/lane schemas; dm_review_adapter.py and receipt translation; tests/test_review_findings.py and adapter tests; plugins/dm-review/skills/review/SKILL.md, commands/dm-review.md, commands/dm-review-fix.md, commands/dm-review-loop.md, and agents/workflow/review-consolidator.md.

  • Write immutable content-addressed records and immediately append their references through the existing crash-safe EventStore; do not invent a second locking protocol.
  • Allow review-owned evidence only under a declared artifact root while forbidding product edits, index/history mutation, PR/issue mutation, and repository tracking artifacts.
  • Move automatic simplification and tracking mutations into explicitly approved fix/loop workflows without losing zero-deferral reporting.
  • Generate Markdown findings and todo-compatible projections from structured authority.

05 — CI evidence and closeout

Primary files: new workflow_kernel/ci_evidence.py and closeout.py, provider-neutral schemas and declared adapter fixtures, tests/test_ci_evidence.py, tests/test_closeout.py, and plugins/dm-review/skills/review/references/issue-tracking.md.

  • Preserve provider-native status/conclusion, subject kind/SHA/ref, check kind, app identity, run/job IDs, attempt, timestamps, URLs, and raw opaque fields.
  • Derive satisfiesProviderMergeRule separately from raw state and only from current repository requirements.
  • Use exact PR head rather than synthetic merge SHA; compare local, reviewed, and delivered state.
  • Separate textual mentions, parsed closing intent, provider-resolved entities, provider closing links, and actual closure; evaluation remains pure and non-mutating.

06 — Assembly repository profile

Primary files: a new Baseplate repository profile under plugins/assembly/skills/assembly-build/references/; agents/workflow/go-test-runner.md, commands/assembly-build.md, and skills/development/setup.md; Assembly plugin dependency metadata; targeted contract fixtures/tests.

  • Encode the live Baseplate Make/Docker contract: Compose run --rm --no-deps app for isolated Go work, -tags=dev, ./cmd/assembly, focused package closure, and declared broader/race/security lanes.
  • Import tests/ux/ task status, personas, routes, viewports, expected outcomes, and screenshot points as scenario declarations rather than prose suggestions.
  • Preserve the browser recovery ladder and keep curl diagnostic-only.
  • Make standalone Assembly build/test invocation use the same planner, requiring a hard Workflow Kernel dependency with a conservative unavailable path.

07 — Pipeline integration and Upstream Improvement Scout

Primary files: new Kernel improvements.py plus schemas/tests; pipeline_adapter.py, dm_review_adapter.py, and translation stages as needed; plugins/pipeline/agents/workflow/execution-orchestrator.md, commands/pipeline.md, commands/pipeline-run.md, references/artifact-lifecycle.md, and references/run-postmortem-schema.md.

  • Use planner and build bindings at dispatch, verification, review, browser, CI, and closeout boundaries; invalidate only affected evidence.
  • Use generated staging allowlists instead of broad directory staging while preserving exact deletion/rename handling.
  • Seal a redaction-safe improvement input index before cleanup; finalize after Docker/artifact/Git cleanup and terminal metrics are available.
  • Candidate authority is structured JSON with stable identity, evidence refs, recurrence, dedupe/completion state, target ownership, acceptance tests, confidence, and safety boundary. The reusable prompt is deterministic generated output.
  • The Scout is proposal-only, runs every time including clean runs, accepts an empty result, and never mutates upstream plugins, routing, findings, releases, or merge authority.

08 — Cross-plugin release integration

Primary files: Kernel cli.py and package exports; all new public schema and CLI inventory tests; tools/validate-workflow-kernel.py, tools/validate-workflow-contracts.sh, tools/check-dependencies.sh, relevant compatibility/runtime tests; canonical Claude plugin manifests and root marketplace; generated Codex manifests/command aliases; CLAUDE.md and plugin reference documentation.

  • Expose stable schema-validating CLI commands with documented exit codes; command execution remains safe argv and provider/network collection stays outside neutral core.
  • Use a coordinated unpublished release set: Workflow Kernel 0.4.0, Assembly 3.9.0, dm-review 1.46.0, and Pipeline 1.33.0, with dependency floors updated together.
  • Regenerate, never hand-edit, .codex-plugin manifests and generated command-skill aliases.
  • Run focused unit suites, Kernel release validation, workflow contracts, generators in check mode, dependency resolution, dual compatibility, full composition, and git diff --check.
  • Separate Depot-owned validation from live sibling integration declarations and preserve exact evidence for any external failure.

Intentional Agent and Human Boundary

Code will own safe command construction, deterministic selection, identity, schema validation, evidence matching, lifecycle/sensitivity facts, provider-state normalization, exact closeout facts, deduplication, and generated views. Agents and humans continue to own ambiguous risk classification inputs, security threat interpretation, architecture and code-quality judgment, UX and visual evaluation, disagreement disposition, improvement prioritization, issue wording, merge recommendation, and release approval.

Verification Strategy

  1. Each chunk runs focused unit and contract tests for its new strict schemas, hostile inputs, legacy defaults, deterministic output, and invalidation behavior.
  2. Consumer chunks prove shadow parity and explicit unavailable behavior before enforcement.
  3. Chunk 08 runs ./tools/validate-workflow-kernel.py, ./tools/validate-workflow-contracts.sh, both generator --check commands, ./tools/check-dependencies.sh, ./tools/validate-dual-compat.sh, ./tools/validate-composition.sh --all, and git diff --check.
  4. No product UI is changed in Depot. Browser verification here is contract/fixture verification; a live Assembly browser run is required only if integration changes Assembly runtime behavior during downstream validation.
  5. No measured savings will be claimed unless the run emits direct before/after usage, duration, wait, setup, or cleanup evidence.
