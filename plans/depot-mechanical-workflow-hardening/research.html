Research Brief: Depot Mechanical Workflow Hardening

Summary

The requested hardening is well aligned with Depot's current direction, but it should extend the existing Workflow Kernel rather than introduce a second harness. PR #12 already supplies content-addressed behavioral contracts, deterministic feedback, stable finding identity, browser recovery, typed cleanup ownership, and attempt economics. The missing layer is deterministic repository-aware verification selection and a set of independently versioned evidence contracts that Pipeline and dm-review can consume.

The recommended architecture is a small contract family: repository verification profile, derived verification plan, lane result, build/evidence binding, append-only finding events, browser scenario and immutable evidence bundle, artifact classification and staging allowlist, CI evidence, closeout audit, and improvement candidates. Workflow Kernel owns neutral validation and transformation; Assembly owns its project defaults; Pipeline and dm-review own orchestration and judgment. New contracts should begin in shadow/compare mode, retain prose fallbacks during migration, and never reinterpret missing legacy fields as a pass.

The Upstream Improvement Scout should run in two deterministic stages. Before cleanup it seals a safe index of candidate evidence and provenance; after cleanup it adds exact cleanup outcomes, deduplicates completed and standing recommendations, emits structured proposal-only candidates, and generates a reusable upstream Pipeline prompt. This preserves evidence that cleanup may delete while still capturing Docker and other cleanup friction. An empty candidate set is a successful, valid result.

Project Context

  • Runtime boundary: Workflow Kernel is the sanctioned Python 3.12 standard-library runtime. The rest of Depot remains Markdown and JSON consumed by plugin hosts.
  • Platform boundary: Claude manifests and commands are canonical. Codex manifests and command-skill aliases are generated and must not be edited directly.
  • Current integration seam: Pipeline and dm-review already call Kernel adapters, compare shadow output, register exact-owned cleanup resources, and consume behavioral verification contracts.
  • Delivery boundary: this run is stacked on PR #12 head 8faeb6ddf00979451c8fd25d7f0bd97f37ae86a8; it must remain an unmerged draft PR and must not publish plugin releases or refresh installed caches.
  • Baseline boundary: Depot workflow contracts pass. The Kernel validator's live sibling-Assembly declaration failure must remain distinguished from Depot-owned release failures.

Assembly Baseplate as the first repository profile

The live sibling repository confirms that project policy cannot be inferred safely from generic Go conventions. Its canonical Make targets use docker compose run --rm --no-deps app, -tags=dev, ./cmd/assembly, focused permission packages, and separate UX index/suite validation. Its CI intentionally excludes test-race and container-scan from pull requests, while scheduled and push lanes carry different authority. These facts directly contradict the stale Assembly plugin prose that assumes docker compose exec app, ./cmd/api, untagged tests, and full race testing for every Go change.

Baseplate also has a markdown-first UX framework with executable task indexing, persona/status declarations, explicit viewports, routes, screenshot points, expected authorization outcomes, and ignored per-run evidence directories. The planner should discover and validate these project declarations rather than invent browser coverage. A profile may reference executable project commands and declaration paths, but commands must be stored as argument arrays and executed without shell interpolation.

Domain Knowledge

Verification planning

A repository profile is policy; a derived verification plan is a decision for one exact change; a lane result is observed evidence. Keeping them separate prevents stale plans from becoming project truth and makes invalidation precise. The profile should declare supported tools, safe command arrays, tag/build/package conventions, change classifiers, risk escalators, required evidence by event scope, UX/persona declaration sources, and ownership. The plan should bind profile digest, repository state, changed paths, risk inputs, selected and omitted lanes, reasons, expected authority, and budget. Results should bind the plan and retain raw exit state, timestamps, artifact references, and unavailable/skipped distinctions.

The tier ladder should be explicit and additive: local static/schema checks; focused package tests; broader project tests; race/security/container work; browser/accessibility/persona scenarios; remote PR checks; and push, schedule, merge-queue, or post-merge proof. A lower tier can reduce repeated local work but cannot substitute for a higher-authority event lane. Sensitive paths and final review retain full fanout.

Review findings and read-only behavior

dm-review already has stable canonical/source finding IDs, raw references, disagreement preservation, provenance, coverage gaps, and a consolidator. The next safe step is content-addressed immutable finding and lane records referenced immediately through Workflow Kernel's existing crash-safe EventStore, with Markdown generated as a view. This reuses the established append/replay protocol instead of creating a second JSONL authority. Partial agent failure should leave valid earlier findings and an explicit coverage gap. Final disposition may supersede prior state but must not rewrite history.

The public dm-review and underlying review workflow should be inspection-only. Automatic simplification edits, staging, commits, todo creation, and issue creation belong only in explicitly approved dm-review-fix or dm-review-loop flows. Read-only enforcement should be mechanically checked through before/after repository state and emitted as a review boundary result.

Build and evidence identity

Evidence becomes stale when relevant source, generated output, command/profile policy, build artifact, image, or environment identity changes. A reusable binding envelope should include exact commit, tracked diff digest, relevant untracked set or explicit exclusion, profile and plan digests, command identity, build/image digest when available, execution timestamps, exit state, and evidence digest. Dirty state is not inherently invalid, but it must be represented rather than hidden. Review receipts should point to the binding instead of restating it.

Artifacts and cleanup

Lifecycle and sensitivity are independent. An artifact can be durable but private, or ephemeral and safe. Classification should record content class, retention class, redaction state, provenance, owner, and whether the path is eligible for staging. The staging allowlist must be generated from approved classified artifacts and exact source paths; broad git add patterns are not an authority. Cleanup should remain exact-owned and guarded, including Docker containers, networks, and volumes created for testing or review.

Improvement Scout

The Scout should code the mechanical half of upstream learning while preserving human judgment. Inputs can include validator failures, retry/recovery events, coverage gaps, repeated setup commands, provider fallback receipts, browser recovery, cleanup outcomes, missing telemetry, existing postmortem recommendations, and known completed controls. Candidate fields should include stable identity, category, proposed upstream owner, evidence refs, recurrence count, confidence, expected benefit, implementation boundary, safety risk, status, and superseded/completed links. The prompt is generated from approved structured candidates; it is never the authority.

Current evidence supports candidates for canonical verification launch commands, Depot-owned versus sibling release boundaries, token/cost and typed-wait telemetry, and a diagnostic for high review iteration counts. It does not support claims of measured token, cost, time, or Docker-cleanup savings. Provider attempt receipts and prior browser recovery controls are already implemented and should be deduplicated rather than re-proposed.

Design References

No product interface is in scope, so no visual design system or screenshot baseline applies. The relevant information-design rule is: code establishes facts; agents interpret facts. Structured artifacts should expose raw state, derivation, authority, provenance, and absence explicitly, while human-facing Markdown and HTML summarize those facts without becoming a competing editable source.

Baseplate's UX task declarations are nevertheless a first-class verification input. The Kernel should preserve persona, route, viewport, expected outcome, steps, assertions, screenshot points, authentication boundary, engine, and human-action pauses. One immutable evidence bundle can then be shared by independent reviewer lanes, avoiding repeated browser setup while keeping accessibility, visual, interaction, and domain interpretations separate.

Technical References

GitHub CI authority

  • GitHub check status and conclusion are separate. neutral and skipped can satisfy dependent jobs or protected-branch rules, but they remain distinct raw conclusions and must not be normalized to passed.
  • Whole workflows skipped by path, branch, or commit filters can leave a required check pending, while a conditionally skipped job reports success. The normalized record must retain check kind, raw provider state, required-policy source, and a separately derived merge-rule result.
  • Pull-request workflows normally run against a test-merge SHA; the actual head is github.event.pull_request.head.sha. Push, schedule, and merge-group events have different subject commits and authority. PR success cannot prove push, scheduled, or post-merge health.
  • Required-check authority may bind a context to an expected GitHub App and strict update policy. A green-looking check from an unapproved source is not authoritative.

Closeout semantics

  • Compare local, reviewed, and delivered commits against the PR's actual head.sha/headRefOid, not merge_commit_sha, which can represent a synthetic test merge.
  • Closing keywords in a PR body only create a closing link when the PR targets the repository's default branch. Plain mentions are references, not closing intent; auto-close can also be disabled by repository policy.
  • Keep parsed references, provider-resolved entities, closing intent, provider closing links, and actual closed state separate. Verify referenced entities and preserve unresolved provider authority rather than guessing.

Provider boundary

No authoritative public Blueprint CI status/API specification was identified, and no Blueprint payload fixture exists in the inspected repository. The implementation must not assume GitHub-compatible semantics. Define a provider adapter contract with opaque raw fields, explicit mapping tables, capability flags, subject identity, stable run/job IDs and URLs, and unavailable/unresolved outcomes when authority cannot be established.

Migration pattern

  1. Add independently versioned schemas, stdlib-only modules, CLI commands, hostile-input tests, and stable inventory assertions.
  2. Add repository profile discovery and Assembly Baseplate defaults without changing current prose authority.
  3. Emit new structured artifacts in shadow mode and compare their selected work, evidence coverage, and generated views with current behavior.
  4. Promote deterministic facts to authority only after parity; keep semantic risk, UX quality, threat interpretation, and merge recommendations agent/human-owned.
  5. Regenerate Codex command aliases and manifests from canonical Claude sources; run Kernel, workflow-contract, dual-compatibility, dependency, and full composition validators.

Constraints and Risks

  • Universal-schema risk: combining profiles, plans, evidence, findings, CI, and closeout into one receipt would couple unrelated lifecycles and make compatibility unsafe.
  • Legacy ambiguity: missing legacy fields must produce explicit default provenance or unavailable, never low risk, passing, or authoritative.
  • Authority collapse: local, PR, push, schedule, merge-group, and post-merge evidence are non-interchangeable. Raw provider conclusions and derived policy acceptance must both survive normalization.
  • Stale evidence: source, generated code, dirty state, profile, selected command, artifact/image, or reviewed commit changes can invalidate different evidence subsets. Invalidation must be dependency-aware.
  • Read-only regression: moving mutations out of review can expose workflows that silently depended on simplification commits or automatic issue creation; contract tests must prove the new boundary.
  • Evidence privacy: browser bundles and logs may contain credentials, private member data, cookies, filesystem paths, or tokens. Redaction must occur before durable promotion and before the Scout indexes evidence.
  • Cleanup ordering: running the Scout only after cleanup can lose raw evidence; running it only before cleanup misses Docker and artifact cleanup outcomes. The two-stage seal/finalize design is required.
  • Provider uncertainty: Blueprint semantics are unknown. A provider-neutral schema must preserve opaque fields and fail unresolved rather than fabricate mappings.
  • Measurement limits: current runs lack durable Codex token/cost counters and typed wait receipts. Only directly observed counts and durations may be reported.
  • Release surface drift: new public schemas and CLI commands require updating exact inventory tests, manifests, generated aliases, documentation, and version numbers together.

Key Decisions Needed

  1. Contract granularity: approve the bounded contract family above rather than a universal receipt. Recommended: yes.
  2. Profile precedence: project-owned explicit config should override plugin defaults; detected conventions may suggest but must not silently override either. Recommended precedence: project config, plugin profile, conservative unavailable.
  3. Finding storage: use immutable finding/lane records and append their references through the existing crash-safe EventStore, plus a deterministic projected snapshot/Markdown view. Recommended: event history and referenced records are authority; views are replaceable.
  4. Browser reuse: capture one immutable runtime bundle per exact scenario/build and fan it out to independent interpretation lanes; re-run only when an assertion requires live interaction or the binding is stale. Recommended: yes.
  5. CI acceptance: preserve provider state and compute policy acceptance separately from live repository rules. Recommended: yes; unknown authority remains unresolved.
  6. Scout placement: seal safe evidence before cleanup and finalize after exact-owned cleanup, including Docker outcomes. Recommended: yes.
  7. Rollout order: build the neutral profile/planner/binding foundation first, then artifact safety, browser bundles, structured findings/read-only review, CI/closeout, Scout integration, and cross-plugin release validation.
